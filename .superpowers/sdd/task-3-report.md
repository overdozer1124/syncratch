# Task 3 implementation report

## Outcome

Implemented a static `apps/editor-web` Scratch editor and the browser-only
`@blocksync/project-store-idb` package.

The app:

- mounts the pinned standalone Scratch GUI generated by the existing Task 0
  vendor build;
- creates the first local project from the existing safe browser fixture and
  restores the newest local record on reload;
- attaches `ScratchStorage` backed by an in-memory asset map;
- converts between `ProjectDocument` and the VM with
  `@blocksync/sb3-tools/browser`;
- marks edits dirty from `PROJECT_CHANGED`, debounces atomic saves, and keeps
  newer edit generations dirty until their own save completes;
- supports New, validated `.sb3` import, `.sb3` download, explicit Save, title
  editing, save status, and retry/export after save failure;
- detects stale revisions without overwriting the newer IndexedDB record;
- collects costume/sound bytes from the VM runtime so newly created assets are
  included in atomic save and export;
- exposes `window.__blocksyncTask3` actions that operate on the real VM,
  conversion, export/import, and persistence paths.

The store uses IndexedDB database version 1 and a `projects` object store keyed
by `localProjectId`. CAS read/compare/write occurs in one readwrite
transaction. All public records are structured-cloned.

## Store coverage

`fake-indexeddb` tests cover:

- create/read/list and detached structured clones;
- atomic expected-revision replacement;
- stale CAS and preservation of the newer record;
- close/reopen reload;
- delete and typed not-found;
- malformed input rejection without a partial record;
- transaction abort without a partial record;
- quota error mapping;
- `Uint8Array` preservation.

- `ProjectStoreNotFoundError` / `NOT_FOUND`
- `ProjectStoreRevisionConflictError` / `STALE_REVISION`
- `ProjectStoreInvalidRecordError` / `INVALID_RECORD`
- `ProjectStoreTransactionError` / `TRANSACTION_FAILED`
- `ProjectStoreQuotaError` / `QUOTA_EXCEEDED`

## E2E acceptance

Playwright runs against `vite build` output served by `vite preview`, not the
development server. It uses the actual standalone Scratch GUI and covers:

1. fresh GUI mount and local save readiness;
2. real VM block creation, autosave, reload, and block restoration;
3. browser-safe export followed by validated import as a new local project;
4. simulated IndexedDB save failure while `.sb3` download remains available;
5. blocked non-static requests after initial load while edit/save/reload stays
   local.

## Known limitations

- A stale-tab conflict is non-destructive and export remains available, but
  automatic merge/rebase is intentionally not implemented in this MVP.
- “Most recently opened” is represented by the record `updatedAt`; there is no
  separate last-opened metadata field in the required `LocalProjectRecord`.
